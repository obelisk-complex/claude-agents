# Claude Code Agents

[![Quillx](https://raw.githubusercontent.com/qainsights/Quillx/main/badges/quillx-4.svg)](https://github.com/qainsights/Quillx) Agent specifications and prompts were generated by Claude under direction. The scope, purpose, security model, and review priorities for each agent were defined by a human. Outputs were reviewed and iteratively refined through real-world use on production codebases.

A collection of specialised [Claude Code](https://code.claude.com/docs/en/overview) sub-agents for requirements engineering, code auditing, QA, security review, performance analysis, platform compatibility testing, accessibility, SEO, design, and red-team assessment.

## Installation

Copy the agent files you want into your project's `.claude/agents/` directory:

```bash
# Copy all agents
mkdir -p .claude/agents
cp agents/*.md .claude/agents/

# Or copy specific ones
cp agents/code-auditor.md .claude/agents/
cp agents/qa-agent.md .claude/agents/
```

For user-scope agents (available in every project), copy into `~/.claude/agents/` instead. Claude Code automatically discovers agents in either location.

## Usage

```bash
# Natural language in a Claude Code session
"Use the code-auditor agent to review src/"

# @-mention
@code-auditor review the auth module

# Session-wide (the agent handles everything)
claude --agent code-auditor
```

When invoked from the main conversation, Claude Code spawns agents as sub-agents with their own context window. Results are returned to the parent conversation.

## Security Model

- **Read-only by default.** Most agents run in `permissionMode: plan` - they can read code and run analysis commands but cannot modify files.
- **Write-capable agents** (`interview`, `qa-agent`, `coverage-analyst`, the `*-test` agents, `travel-*` writers, `stitch-designer`, `agent-auditor`) use `acceptEdits` for spec files, tests, reports, or agent definitions.
- **qa-agent uses worktree isolation.** It has `acceptEdits` permission but operates in a temporary git worktree, so changes don't affect your working tree until you review and merge them.
- **No MCP dependencies.** Fleet agents use only built-in Claude Code tools (Read, Write, Edit, Grep, Glob, Bash, WebSearch, WebFetch, Agent). External MCP servers are treated as threat vectors and excluded by design.
- **Web tools.** Agents with `WebSearch` / `WebFetch` can query CVE databases, documentation, and known-issue trackers. Each web-using agent is instructed to consult a local project knowledge base first (e.g. `llm-wiki/`, `wiki/`, `docs/research/`) before fetching externally, and to redact project-specific identifiers from queries.
- **Memory.** Agents with `memory` enabled accumulate knowledge across conversations in `.claude/agent-memory/` (project scope) or `~/.claude/agent-memory/` (user scope).

## Agent Design Principles

- **Find real problems, not style nits.** Every finding should be something that matters.
- **Concrete fixes, not vague suggestions.** Include file paths, line numbers, and code snippets.
- **Severity classification.** Critical/High/Medium/Low/Info so teams can prioritise.
- **No manufactured findings.** If the code is clean, say so.
- **Aggressive verification.** Agents cross-check against CVE databases, current documentation, and the project's own prior research before reporting.
- **Domain principles plus cross-fleet principles.** Every agent's `Guiding Principles` section splits domain-specific rules from cross-fleet rules (warnings are errors, leave no trash, do the harder fix, secure by default) so behaviour stays consistent across the fleet.

## Model Variants

All agents default to `sonnet`. Opus variants are provided for higher-reasoning tasks where the extra depth is worth it.

| Directory | Model | Use when |
|-----------|-------|----------|
| `agents/` (default) | `sonnet` | Standard use. |
| `agents/opus-variants/` | `opus` | Maximum reasoning depth is worth the cost (available for `agent-auditor`, `blind-spot-auditor`, `conformance-auditor`, `interview`, `migration-planner`, `plan-auditor`, `requirements-auditor`, `travel-research-coordinator`). |
| `agents/sonnet-variants/` | `sonnet` | Scoped variants with tighter turn/effort budgets (available for `agent-auditor`, `blind-spot-auditor`, `migration-planner`). |

To switch an agent to a variant, copy the variant file over the default:

```bash
# Example: use the Opus variant of conformance-auditor
cp agents/opus-variants/conformance-auditor-opus.md agents/conformance-auditor.md
```

## Customisation

Each agent is a Markdown file with YAML frontmatter. You can:

- Adjust `maxTurns` to give agents more or less time.
- Change `model` (`sonnet`, `opus`, `haiku`) to control depth vs cost.
- Set `effort` to `low`, `medium`, `high`, or `max` to control reasoning depth.
- Add `memory` (`user`, `project`, or `local`) for cross-session learning.
- Set `color` for visual identification in the UI.
- Add `skills` to preload domain knowledge.
- Add project-specific context to the prompt body.
